## 1.  현재 우리가 쓰는 언어모델
대부분 __autoregressive__ 모델임
확률의 연쇄법칙을 그대로 적용한 것. 
다음단어를 계속 예측하는 것이 모델의 핵심

### 언어모델의 작동방식
1. 각 단어들을 토큰화해서 각각 ID를 부여
2. ID들을 신경망에 통과시킴
3. 다음 토큰에 대한 확률 분포를 출력
4. 거기서 샘플링해서 새 토큰 선택
5. 다시 역토큰화해서 단어로 변환
### 언어모델을 훈련할  때 중요한  5가지
1. 아키텍처 . 어떤 구조 신경망을 쓸 것 인가 ?
2. 훈련 손실 및 알고리즘 . 어떻게 모델을 최적화 할 것 인가 ?
3. 데이터 . 뭘 학습시킬 것 인가?
4. 평가 . 모델이 실제로 잘하고 있는지 어떻게 알 것인가?
5. 시스템 . 거대한 모델을 현실 하드웨어 위에서 어떻게 돌릴 것인가 ?

## 2. 토크나이저
왜 필요한건가? 
- 오타, 신조어, 공백 없는 언어를 처리못함.
- 시퀀스가 너무 길어짐, 트랜스포머는 시퀀스 길이의 제곱에 비례해서 연산증가
### 토크나이저가 왜 중요한가 ? 
- 파이썬에서 들여쓰기에 쓰는 공백 네개가 이전에는 이상하게 처리되었는데 이기능이 코드를 방해함. 327같은 숫자가 한의 토큰으로 처리되면 모델은 추리처럼 각 자릿수를 분리해서 생각할 수 없어짐
### BPE : Byte Pair Encoding
가장 많이 쓰이는 알고리즘
1. 큰 컨텍스트 코퍼스로 시작
2. 처음엔 모든 문자에 각각 토큰을 부여
3. 가장 자주 등장하는 토큰 쌍을 찾아서 합침
4. 반복
- 공백 이나 구두점 처리는 사전 토크나이저가 먼저 처리

## 3. 프리트레이닝 벤치마크 방식
### Perplexity (언어모델의 대표적 평가 지표)
``` Perplexity = 2^(평균 토큰당 손실) ```

1. 검증 손실을 조금 더해석하기 쉽게 변한한 것
  - 평균 토큰 당 손실을 구하고 거기다 지수를 취함
    - 왜 지수를 취하는가? 로그는 사람이 직관적 이해 어려움
 2. 모델이 다음 단어를 예측할 때 몇개의 단어 사이에서 망설이는지를 나타냄 
    - Perplexity = 1 이면 완벽한 모델이다. 항상 정확한 단어를 예측할 수있다
    - Perplexity = 어휘 크기라면 완전 무작위 상태, 아무것도 모르는 상태

    ### 학계 벤치마크로 잘 사용하지 않는 이유
    1. 토크나이저에 따라 값이 달라짐
    2. 어휘 토큰의 개수에따라 상한 자체가 달라져 단순비교가 불가능
    3. 현재는 MMLU같은 벤치마크를 개발단계에서 사용
	- 개발 과정에서는 여전히 사용되지만 공식 비교에선 사용하지 않음.

### MMLU 벤치마크
가장 많이 쓰이는 공식 (대학의학, 물리학, 천문학, 다양한 분야의 4지선다 문제로 구성)
    ### 평가방법 
    1. 네가지 선택지 각각이 나올 확률을 보고 높은것을 선택
    2. 모델에게 A, B, C, D 중 하나를 고르라고 하고 다음 토큰이 뭔지 봄

- 평가의 함정이 존재함
 같은 MMLU도 평가 방법에 따라 결과가 크게달라짐

### 벤치마크 오염문제
-모델이 벤치마크 테스트셋으로 학습했을 때 우리는 무엇으로 학습했는지 알 수 없다.
모델이 정말 자신의 추론능력으로 문제를 해결한 것인지, 답안지나 특정 시험의 답을 암기력으로 통째로 암기하여 이런 성능이 나오는 것인지.
- 탐지방법: 대부분의 데이터셋은 순서가 있음, 모델이 특정 데이터 셋에 성능이 뛰어나거나 데이터 셋을 학습하면서 데이터셋을 전부 삼켰는지 알고자하면 LLM 특징을 다음 단어 예측을 역이용한다. 
- 모델에게 데이터셋 예시들을 원래 순서대로 생성하게 하는 것 vs 섞인 순서로 생성하게 하는 것을 비교. 원래 순서로 더 잘 생서앟면 그 순서 자체를 기억하는 것으로 훈련에 포함되었다는 신호다.

## 4. 데이터
### 대규모 언어모델이 인터넷에서 어떻게 학습하는가 ?
1. __1단계. 인터넷 전체다운로드__ 
    - Common Crawl 같은 웹크롤러 사용
    - 약 2500억개 페이지, 1페타바이트 데이터 6기가 바이트 짜리 압축 파일로 배포
2. __2단계. HTML에서 텍스트 추출__
    - HTML 태그 제거,반복요소 제거 수식 추출이 어려움
3. __3단계. 불필요 콘텐츠 필터링__
4. __4단계. 중복제거__
5. __5단계. 휴리스틱 필터링__
    - 저품질 문서 제거
    - 토큰 분포가 일반적인 것과 너무 다른 문서
    - 단어 길이가 너무 극단적인 문서
    - 단어가 세 개밖에 없는 문서
    - 단어가 1,000만 개인 문서
6. __6단계. 모델 기반 필터링__
    - 위키피디아에서 참조하는 링크들을 모음
    - 위키피디아를 기준으로 분류기를 학습시켜 품질 좋은 문서를 학습
7. __7단계. 도메인 분류 및 가중치 조정__
    - 데이터를 *엔터테이먼트*, *책*, *코드* 등으로 분류하고 가중치를 조정
    - (코드로 더 많이 학습시키면 추론능력이 올라간다고 알려져있다.)
8. __8단계 고품질 데이터로 파인튜닝__
    - 학습 마지막 단계에서 학습률을 낮추고 위키피디아 같은 고품질 데이터로만 학습을 진행함, 일종의 오버피팅으로 의도적으로 진행
    ### 왜 오버피팅을 하는가 ? 
	크게 3가지의 이유가 존재한다고 함
    1. Sanity Check
	    데이터가 적은데도 오버피팅이 불가능하다면, 모델 구현 자체가 논리적으로 틀렸거나 데이터가 모델에 제대로 전달되지 못하다는 증거가 될 수있음 
    2. Specialize Tasks
        새로운 데이터에 대한 대응력보다 기존 데이터의 완벽한 암기력에 대한 성능 지표확인하기 위해 
    3. Double Descent 
        최신 인공지능에서 오버피팅의 임계점을 넘어 모데을 극단적으로 복잡하고 파라미터의 수가 늘어난다면 초반에 성능이 나빠지다가 파라미터가 데이터 수보다 압도적으로 많아지는 지점을 넘어서면 모델이 더 똑똑해진다는 현상이 관찰됨 (모델은 클수록 좋다) 
### 데이터 규모 변화 
     - 초기 학술 모델 : 약 1500억 토큰(800GB)
     - 현재 최고 모델(SOTA) : 약 15조 토큰 (55TB~60TB)

## 5. 스케일링법칙 
- 데이터가 많을수록, 모델이 클수록 예측이 가능한 방식으로 좋아짐 (선형로그 방식으로)
- 모델을 크게 만들수록 성능이 계속 좋아지고 오버피팅이 없어진다 
    ### 스케일링 법칙의 형태
    - 컴퓨팅-손실, 데이터-손실, 파라미터-손실을 각각 로그-로그 스케일로 그리면 선형 관계가 그려진다. 즉 로그-로그 스케일에서직선이다. 컴퓨팅을 10배 느릴면 손실이 얼마나 줄어드는지 예측 할 수 있다는 뜻이된다.
        #### 이 포인트가 중요한 이유
        - 2~3년 후 모델이 얼마나 좋아질 지 예측이 가능하다. 컴퓨팅이 계속 증가할 거라는 걸 알면, 성능도 예측 가능하게 좋아질 것.
### 스케일링 법칙의 실용적 활용

예를 들어 GPU 1만개를 한달동안 빌렸다고 가정했을때 

- 과거방식
    - 30일 중 29일짜리 모델을 1일짜리로 30번 실험해서 제일 좋은걸 골라쓴다 -> 최종모델이 하루 치 연산량만 들어간 꼬꼬마 모델이 탄생한다. 나머지 29일치 막대한 GPU파워는 어떤 세팅이 좋은지 간을 보는데만 쓰고 허공으로 날라가버리는 상황이 발생 

- 새 방식(스케일링 법칙)
1. 작은 모델들을 여러크기로 학습시켜 스케일링 법칙 피팅
2. 작은 모델에서 하이퍼 파라미터 튜닝 
3. 큰 모델로 성능 외삽
4. 27일짜리 큰 모델을 학습
    
- 스케일링 법칙으로 보면 대부분의 아키텍처 차이함수 기울기는 거의 같고 절펴만 조금 다르다. 그 차이는 GPU를 조금 더 사용하거나 다음 세대 GPU를 기다리면 사라진다. 
__즉 아키텍처에 너무 집착할 필요는 없다__ 
__반면 데이터는 이야기가 다르다. 좋은 데이터를 쓰면 스케일링 버칙 자체가 훨씬 좋아진다__

### Chinchilla(친칠라)법칙
스케일링 법칙을 이용해 최적의 자원 배분을 찾는 것.

__Q: 같은 컴퓨팅으로 더 큰 모델을 적게 학습시키는게 좋을까, 작은 모델을 많은 데이털 학습시키는 게 좋을까 ?__

__Chinchilla 논문의 실험 방법:__
1. 같은컴퓨팅으로 파라미터 수와 토큰 수를 조합해서 여러 모델 학습
2. 각 컴퓨팅 예산에서 최적의 조합을 찾음
3. 이걸 다시 스케일링 법칙으로피팅

__결론: 파라미터 1개당 토큰 20개가 학습의 최적값이다__
- 단 이건 학습 비용 최소화 기준임. 실제 서비스에서는 추론 비용이 존재함. 추론 비용까지 고려한다면 __150 토큰__ 장도가 현실적 최적값.

## 6. 포스트트레이닝 
### 포스트트레이닝이 왜 필요한가 ?
프리트레이닝만 된 GPT에게 "달 착륙을 6살 아이에게 설명해줘"
라고하면
```
달 착륙을 6살 아이에게 설명해줘  
달 착륙 이론을 8살 아이에게 설명해줘
```
인터넷에서 질문 옆에는 대체로 비슷한 질문이 또 있으므로, 모델이 그 패턴을 학습해서 답 대신 질문을 생성해버린다.

- 포스트 트레이닝의 목표:
	1. 지시 따르기: 사용자가 원하는 걸 실제로 해주기
	2. 모더레이션: 유해하거나 독성 있는 내용 거부하기

## 7. 지도 파인튜닝 (SFT)
인간이 만든 (질문, 이상적인 답변) 쌍을 수집해서, 이 데이터셋을 언어 모델에 파인튜닝한다. 손실 함수는 프리트레이닝과 동일하게크로스 엔트로피. 차이는 데이터가 일반 인터넷이 아니라 "질 높은 원하는 답변"이라는 점.

### Alpaca: :LLM으로 SFT 데이터 만들기

1. 사람이 만든 175개의 (지시,답변) 쌍 준비
2. GPT-3 text-davinci-003한테 "이런거 52,000개 더 만들어줘"
3. 이렇게 만들어진 데이터셋을 원하는 모델에 파인튜닝
이게 합성 데이터의 시작점

### SFT의 핵심
데이터가 많이 필요하지않음. LIMA 논문에 따르면 2,000개에서 32,000개로 데이터셋을 늘려도 성능차이가 별로 없다는 것이다.

- 이유 SFT는 새로운 지식을 배우는 게 아님. 모델은 이미 프리트레이닝에서 인터넷을 다 보았다. SFT는 수많은 사용자 유형 중에서 "이 휴영의 사용자처럼 답변 해."라고 알려줄 뿐이다.

### SFT의 문제점
1. 인간 능력에 제한됨
	- 좋은 글을 알아보는 능력이 좋은 글을 쓰는 능력보다 훨씬 뛰어난 경우가 많음. SFT는 인간이 직접 쓴 답변을 그대로 따라하므로 이 한계에서 벗어나지 못함
2. 환각의 원인
	- SFT 데이터에서 인강니 특정 책을 참고 문헌으로 적었는데, 모델이 프리트레이닝에서 그 책을 본 적 없다면? 모델 입장에서는 "그럴듯하게 들리는 뭔가를 생성하라"는 신호로 받아들임
3. 비용
	- 사람이 직접 고품질 답변을 쓰는 건 엄청나게 비싸다

## 8. RLHF(Reinforcement Learning Human Feedback): 인간 피드백을 통한 강화학습
### SFT와 RLHF의 비교 
	- SFT: "인간이 생성한 답변을 그대로 모방"
	- RLHF: "인간이 더 선호하는 답변이 나오도록 최적화"

인간이 직접 쓰지 않아도, 어느게 더 좋은지 고르기만 해도 된다.

#### 파이프라인:
1. 하나의 지시에 대해 모델이 두 개의 답변 생성
2. 사람이 두개의 답변 중 더 선호하는 쪽을 선택
3. 더 선호하는 답변이 많이 나오도록 모델 업데이트

### 보상 모델
이진 보상 (+1/-1(매핑해서 사용))은 "조금 더 좋아." vs "훨씬 더 좋아."를 구분하지 못함. 그래서 보상모델을 따로 분리하여 학습시킴

회귀모델
```
R(입력, 출력) -> 실수 값(보상 점수)
```

Bardley-Terry 모델
```
P(답변 A가 답변 B보다 좋을 확률)= sigmoid(R(A) - R(B))
```
보상 모델이 얼마나 더 좋은지를 연속값으로 표현함.

### PPO(Proximal Policy Optimization)
강화학습의 알고리즘 중 하나로 모델의 학습 안정성 확보가 목표
- 강화학습 과정에서 언어모델이 더 높은 보상을 받기 위해 파라미터를 업데이트할때, 한 번에 너무 크게 번형되어 기존의 언어 능력이 붕괴되는 현상을 막음.
- 모델의 가중치의 업데이트를 보폭을 수학적으로 강제 제한하여 모델이 망가지지 않는 선에서 점진적으로만 성능을 개선하도록하는 최적화 알고리즘.

#### 예시(OpenAi가 처음 쓴 방법)
```
1. SFT 모델 훈련
2. STF 모델로 답변 생성 -> 인간에게 비교 피드백 수집
3. 보상 모델 훈련
4. PPO로 보상 모델 점수 점진적 최대화
5. 새로운 데이터 수집 -> 반복
```
#### 문제점: 강화학습 자체가 구현하기 매우 복잡함

### DPO(Direct Preference Optimization): PPO의 단순한 접근
별도의 보상모델 없이 선호도 데이터를 직접 언어모델의 Loss에 반영하여 최적화

<img width="643" height="56" alt="Image" src="https://github.com/user-attachments/assets/74bcb043-0c09-45f8-ada2-d525a3329154" />

- 좋은 답변이 나올 확률은 최대화하고 나쁜 답변이 나올 확률을 최소화함

PPO 모델은 답변 생성-> 보상 모델 훈련-> 강화학습이 필요하지만. DPO는 최대 우도 추정 한 번으로 훨씬 단순하고 성능도 같게 처리가 가능하여 __현재 오픈소스 커뮤니티와 산업 표준이다__

### 기존 지표가 안되는 이유
1. validation loss: PPO를 거친 모델과 DPO 거친 모델은 비교가 불가능함
	- 두 모델이 같은 것을 잘하도록 훈련된 게 아니기 때문에, 무엇을 기준으로 비교할지 자체가 불명확하다.  
2. Perplexity: 프리트레이닝을 끝낸 언어모델에게는 훌륭한 성적표가 되지만, 사람의 입맛에 맞추는 얼라인먼트 튜닝을 거치는 순간 퍼플렉서티를 쓰는 것은 틀린 접근이다
3. 평가 자동화는 개방형에는 적절치 못함
	- 사람이 던지는 질문은 텍스트 생성부터 요약까지 맥락에 따라 정답이 수만 가지로 갈리는 개방형이 대부분이다 이렇게 무한한 스펙트럼의 주관적이고 개방적인 결과물을 기계가 무슨 수로 단일 잣대를 들이댈 수 있는가 ?

## 11. GPU에서 모델을 어떻게 효율적으로 돌리나

### GPU 101

|     |    CPU    |      GPU       |
| :-: | :-------: | :------------: |
| 최적화 |    지연     |      처리량       |
| 코어  |   적고 복잡   |    수천 개, 단순    |
| 강점  | 복잡한 단일 작업 | 같은 연산 대규모 병렬처리 |

GPU는 행렬 곱셈에 최적화되어있다. 딥러닝에서 뭔가를 행렬 곱으로 표현할 수 있으면 다른 것보다 10배 빨라진다

LLM 연산: 수십억 개의 파라미터를 동시에 계산해야 하므로, 개별 속도는 느려도 머릿수가 압도적인 GPU가 절대적인 우위를 차지한다

__양자화와 메모리 계산__
- FP16(표준모델): 파라미터당 2바이트 필요 
  (예: 7B 모델 $\approx$ 14GB VRAM)
- 4-bit 양자화: 파라미터당 0.7~0.8바이트 필요
  (예: 7B 모델 $\approx$ 5~6 VRAM)

|          지표          | 의미             | 역할                                                 |
| :------------------: | -------------- | -------------------------------------------------- |
|         VRAM         | GPU의 <br>메모리용량 | 가장 중요함. <br>모델이 메모리에 적재되는지를결정                      |
|        Memory        | 메모리 <br>대역폭    | 데이터를 연산장치로 전달하는 속도<br>토큰 생성 속도를 결정함                |
| Compute<br>Teraflops | 연산 성능          | 실제 계산 속도, FP16, BF16, INT8 등 데이터 정밀도에 따른 연산 능력을 체크 |

__핵심 병목:__ 컴퓨팅 속도 > 메모리 대역폭 > 통신속도. 잘 최적화된 코드에서도 모델 플롭 활용률 50%면 잘 하는 것

### 저정밀도 학습
32비트 대신 16비트를 쓰면 메모리 전송량이 즐어들고 속도가 빨라짐.

딥러닝에서는 소수점 이하 정밀도가 크게 중요하지 않다. SGD 자체가 이미 노이즈가 많은 과정이라서 0.01이냐 0.015냐 차이가별로없다
- 여기서 __SGD__ 란? : 수학적으로 __확률적 경사 하강법__ 이라 불림 LLM 학습의 핵심으로 모델 학습의 궁극적 목표인 오차가 가장 적은 최적의 파라미터를 찾는 것.
- __노이즈가 많다는 것은 무엇인가?__ : 소음이 아닌 방향의 불확실성을 뜻함. 어떤 데이터를 무작위로 뽑았느냐에 따라 매 스텝마다 계산된 방향이 요동치는 것. 너무 정직하게 직선으로 내려간다면 local minima를 만났을 때 Global Minima으로 착각하여 학습을 멈춰버릴 수 있다. 반면 노이즈는 계속 모델을 흔들어대서 local minima를 만났을 때 튕겨져 나오게 만들어주고, Global minima(최종목표)를 찾을 확률이 비약적으로 높아진다

__자동 혼합 정밀도(AMP) 표준 방식__
- 연산: 16비트 (빠름)
- 가중치 저장: 32비트 (정확)

가중치를 32비트로 저장하는 이유: 학습률이 매우 작을 때 16비트로 업데이트값이  반올림되어 0이 될 수 있어 가중치가 아예 변하지 않는 문제가 생긴다
